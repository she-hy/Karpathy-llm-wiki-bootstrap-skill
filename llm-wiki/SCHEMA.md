- Preserve the source's claim before adding critique or operational interpretation.

## Operations

Before any operation, check for `EXTEND.md` preferences using the priority order in the skill. This wiki includes `.llm-wiki-bootstrap/EXTEND.md` with `bm25.mode: required`, so BM25 should be treated as the default local navigation layer whenever the files are available.

### Ingest

Trigger: user adds a file to `raw/` and says `ingest {filename}`.

Protocol:

1. Read the source file completely.
2. Detect the source language. Write derived content in that language; keep YAML keys, filenames, page types, and structural table headers in English.
3. If BM25 is available, search existing pages before creating new concept, entity, comparison, or synthesis pages.
4. Create or update `wiki/sources/{slug}.md`.
5. Update relevant entity and concept pages.
6. Update `wiki/concept-table.md` for every durable concept created, renamed, merged, split, deleted, or materially revised.
7. Update `wiki/index.md`.
8. Review and revise `wiki/overview.md` when the source changes the big picture.
9. Append to `wiki/log.md`.
10. Rebuild BM25 with `python3 scripts/wiki_fts.py build` and verify with `stats`.

### Query

Trigger: user asks a question about the wiki's domain.

Protocol:

1. Read `wiki/index.md`; for conceptual, relationship, taxonomy, or landscape questions, also read `wiki/concept-table.md`.
2. Run `python3 scripts/wiki_fts.py stats`; if `Fresh: no`, run `python3 scripts/wiki_fts.py rebuild`.
3. Run one or more focused searches with `python3 scripts/wiki_fts.py search "{query}" --limit 10`.
4. Open returned `page_path` files and read full context. Do not answer from snippets alone.
5. Follow relevant wikilinks.
6. Answer with citations to wiki pages, not to BM25 rows or exported chunks.

### Lint

Trigger: user says `lint`, `health check`, or `review wiki`.

Protocol:

1. Read `wiki/index.md`, `wiki/concept-table.md`, and every listed page.
2. Check for broken wikilinks, index drift, concept table drift, stale overview, missing backlinks, weak single-source areas, and BM25 freshness.
3. Report findings by severity.
4. Fix only approved issues unless the user explicitly asks for automatic repair.
5. Update log and rebuild BM25 when pages change.

## Search Layer

BM25 is enabled by default in this generated wiki. It is local, rebuildable, and never a source of truth. It helps the LLM find candidate wiki passages; the wiki pages remain the citation targets.

Normal Query means LLM question answering: check freshness with `stats`, rebuild stale indexes when possible, use `search`, open returned `page_path` files, read full wiki context, and cite wiki pages. Export Use means operations, audit, migration, debugging, or external analysis; do not use exports as the normal query path when `search` is available.

Search output fields:

| Field | Meaning | Use |
| --- | --- | --- |
| `page_path` | Wiki page path containing the match, relative to this wiki root and including the `wiki/` prefix. | Open this page before answering. |
| `heading_path` | Heading trail within the page. | Locate the relevant section. |
| `chunk_id` | `{page_path}#{ordinal}` diagnostic identifier. | Do not cite. |
| `score` | Query-local SQLite FTS5 BM25 score; lower is more relevant and values may be negative. | Rank candidates within one query only. |
| `snippet` | Truncated chunk preview. | Never answer from snippet alone. |

Never cite `chunk_id`, `score`, `indexes/fts.sqlite`, or `exports/*` as evidence in user-facing answers. BM25 data finds candidate wiki pages; wiki pages are the evidence layer.

The BM25 data model is documented in `indexes/README.md`. Its exported rows have exactly these fields:

```text
chunk_id, page_path, title, type, heading_path, ordinal, sources, tags, updated, text
```

## Index Protocol

`wiki/index.md` is the LLM's primary navigation tool. It catalogs all pages by type and points to `wiki/overview.md` and `wiki/concept-table.md` as core maps. Update it on every ingest or structural page change. Keep entries sorted alphabetically within each section.

## Concept Table Protocol

`wiki/concept-table.md` is the LLM's compressed concept map. It complements `wiki/index.md`: the index catalogs pages, while the concept table explains what the concepts mean, how they relate, and what needs maintenance.

Maintain this structure:

```markdown
## Maintenance Rules

## Concept Clusters
| Cluster | Concepts | Current interpretation |
| --- | --- | --- |

## Concepts
| Concept | Working definition | Role in this wiki | Sources | Related pages | Status | Maintenance note |
| --- | --- | --- | --- | --- | --- | --- |
```

Rules:

- Keep one row for every durable concept page under `wiki/concepts/`.
- Update the row whenever a concept page is created, renamed, deleted, merged, split, or materially revised.
- Keep rows sorted alphabetically by concept name.
- Keep definitions concise and evidence-aware; link to the full concept page for detail.
- Use `Status` values such as `high confidence`, `single-source`, `tentative`, `needs sources`, or `contradicted`.
- Column headers stay English as protocol identifiers; row content follows the language of the relevant concept page.

## Log Protocol

`wiki/log.md` is append-only. Every operation adds an entry. Format:

```text
## [YYYY-MM-DD] {operation} | {subject}
- {details as bullet points}
```

Parseable with: `grep "^## \[" wiki/log.md | tail -10`

## Conventions

- Filenames: lowercase, hyphens, no spaces. Example: `persistent-compounding-artifact.md`.
- One concept per page. Split pages that drift into two distinct durable ideas.
- Prefer tables for structured comparisons.
- When uncertain, note uncertainty explicitly rather than omitting it.
- In `sources` frontmatter, use filenames only, not full paths.
- Treat the current wiki as single-source unless future ingest adds corroboration.
